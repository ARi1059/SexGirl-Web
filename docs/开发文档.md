## 1. 项目概述

### 1.1 背景

面向**全球私域流量**（约几万访客）的定制化商品展示 H5。访客浏览精致的商品画廊，看中定制款后，通过页面提供的**微信 / QQ 联系方式**与我方工作人员私聊沟通成交。成交沟通发生在微信/QQ 内，**页面本身不含购物车、下单、支付**。

### 1.2 核心特征

1. **图片密集型展示**：以商品图片为核心，性能瓶颈在图片与首屏加载。
2. **配置驱动**：商品标签内容/条目由管理员在后台高度自定义，前端按配置渲染，而非写死。
3. **动态状态**：每个商品有管理员可随时编辑的动态字段（上下架、今日可接单、自定义状态文字），数据每天可能变化。
4. **双端适配**：同时适配 PC 与移动端。
5. **引流闭环**：每个商品可绑定客服联系方式（微信号 / 微信二维码 / QQ 号 / QQ 二维码），引导加好友。
6. **多人协作**：多名工作人员各自登录后台，超管可管理账号。
7. **可选会员与收藏**：访客可注册登录（自设用户名+密码），收藏心仪商品，在个人中心查看收藏；**浏览本身仍免登录**，仅收藏/个人中心需登录。客户账号与管理员账号完全隔离。

### 1.3 关键诉求

- 优秀的 UI 界面（精致、有动效）。
- 高效的响应速度（全球访问秒开）。
- 管理员可视化后台，无需技术介入即可上下架、设置今日可接单（含可接单文案）、编辑状态文字、维护商品类型与富文本描述。
- 多管理员协作，账号可由超管集中管理。
- 访客可选注册登录并收藏商品；收藏在个人中心集中查看，体验轻量、不引入下单/支付。

## 2. 技术选型

| 层 | 选型 | 作用 / 理由 |
|---|---|---|
| **框架** | Next.js (App Router) + TypeScript | 前台 + Payload 后台 + 数据层同进程一体，一个项目搞定 |
| **内容 / 后台 (CMS)** | **Payload CMS 3.x**（Next 原生集成，同进程） | 自动生成可视化后台、CRUD、字段校验、Lexical 富文本、上传、关系、访问控制——一次性替代自研后台 + Auth.js + Prisma |
| **渲染策略** | ISR + on-demand revalidation | 商品内容静态缓存走 CDN 极速；Payload `afterChange` hook 在保存时触发 `revalidatePath`，秒级生效 |
| **数据库** | PostgreSQL（**Supabase** 托管） | 免运维；Payload 用 Postgres adapter 直连，**自管 schema 与迁移** |
| **数据访问** | Payload **Local API**（读）+ Payload 后台 / Local API（写） | Server Components 直接调用 `payload.find(...)`，无 HTTP 跳转，快且类型安全 |
| **前台 UI** | Tailwind CSS + shadcn/ui | 精致、PC/移动响应式、可深度定制 |
| **动效** | Motion (Framer Motion) + View Transitions | 画廊入场 / 列表↔详情过渡 / 弹层动效 |
| **富文本** | Payload **Lexical Editor** | 商品描述等富文本 / 区块编辑 |
| **后台鉴权** | Payload **内置鉴权**（`users` 集合 + 基于 `role` 的 access control） | 仅 `/admin` 需登录；多管理员 + 超管；前台**浏览**免登录 |
| **前台会员 / 鉴权** | Payload **第二个 auth 集合** `customers`（`loginWithUsername`，与管理员 `users` 完全隔离） | 可选客户账号：用户名+密码注册登录；收藏 / 个人中心需登录；客户**不得**进入 `/admin` |
| **收藏** | `favorites` 关联集合（`customer ↔ product`，仅增删） | 登录客户收藏商品，个人中心查看；归属强制绑定当前用户 + 唯一索引防重复 |
| **商品分类** | `categories` 集合（可复用）+ `products.category` 关系 | 「商品类型」结构化、前台可筛选；改名 `afterChange` 触发 revalidate |
| **图片** | 对象存储（Supabase Storage，S3 兼容）+ `@payloadcms/storage-s3` + Payload upload + `next/image` | 上传 + 自动 WebP/AVIF / 多尺寸 / 懒加载 / 占位 |
| **配置驱动** | Payload **Blocks** / 条件字段 + 前台 `<TagRenderer>` / `<ContactRenderer>` | 标签 & 联系方式按 type 分发，源于 Payload 字段定义，前后台共用自动生成类型 |
| **部署** | Vercel + Supabase | 全球边缘 CDN |

> i18n：先做中文。Payload 自带 localization，文案与字段都可后续按需开多语言，不返工（见决策 #3）。

### 2.1 为什么不是纯静态（Astro / 纯 SSG）

早期方案曾考虑纯静态站，但"**今日可接单**""**自定义状态文字**"这类字段**每天都会变**。纯静态意味着管理员每次改状态都要重新构建部署、等待数分钟才生效，体验不可接受。因此采用**带后台的轻量全栈应用**：不常变的内容（图片、标签）静态化走 CDN，高频字段实时获取，管理员一改前台秒级生效。

### 2.2 为什么用 Payload CMS，而非自研后台 + Auth.js + Prisma（v1.1 决策）

原计划要手写整套 `/admin`：表格、表单、字段校验、登录鉴权、多角色权限、图片上传、富文本编辑——工作量大且属重复造轮子。**Payload CMS 3.x** 一次性提供这一切，且：

- **Next 原生、同进程**：直接装进同一个 Next App Router 项目，不引入独立服务。
- **同一个 Supabase Postgres、单一数据源**：用 Postgres adapter 连库、自管 schema 与迁移，不会像 Sanity 那样多出第二套数据源与同步成本。
- **前台不牺牲性能**：前台用 **Local API** 在 Server Component 直接读库，无网络跳转，照样配合 ISR / CDN。
- **天然满足多管理员 + 超管**：`users` 集合 + access control 即可（决策 #2）。
- **满足"额外 CMS"诉求**：强可视化编辑、Lexical 富文本、区块化内容都内置（决策 #4）。

## 3. 系统架构

```
访客(PC/移动) ──全球CDN──▶ Next 前台(site) ──Payload Local API──▶ PostgreSQL(Supabase)
                            (ISR 静态缓存)                            ▲
                                                                     │ 写入
管理员 ──登录(Payload Auth · users)──▶ Payload 后台 /admin ───────────┤
   上下架 · 今日可接单(含文案) · 状态文字 · 富文本 · 商品&标签&分类&客服&账号管理
        └── afterChange hook 触发对应前台页 revalidatePath（秒级全球生效）          │
客户 ──注册/登录(Payload Auth · customers)──▶ 个人中心 /me + 收藏(favorites) ──────┘
   浏览免登录；收藏 / 个人中心需登录（每用户动态，不走 ISR）
图片 ── Payload upload ──▶ 对象存储(Supabase Storage, S3兼容) + next/image 优化
```

### 3.1 三个关键设计决策

1. **既快又实时**：商品图/标签等不常变内容做 ISR 静态缓存、走全球 CDN；上下架/今日可接单/状态文字等每天变的字段，靠 Payload `afterChange` hook 在保存时自动 `revalidatePath` 对应商品页——访客永远看到最新状态，又不牺牲加载速度。
2. **前台浏览免登录、后台需登录、会员功能需登录**：访客凭链接直接浏览（私域引流零门槛）；`/admin/*` 由 Payload 内置鉴权（`users`）保护，无需额外 Auth.js 或自写 middleware。新增**可选客户账号**（第二个 auth 集合 `customers`，用户名登录）：仅收藏与个人中心需登录，浏览不受影响；客户与管理员两套鉴权（会话/cookie/权限）完全隔离，access 函数按 `req.user.collection` 区分来源。
3. **单一数据源 + 配置驱动**：Payload 集合定义既是数据库 schema 又是后台表单，`payload generate:types` 产出的类型前后台共用；标签用 Blocks、联系方式用条件字段，新增类型只加定义与一个前台渲染分支。

## 4. 数据模型（Payload Collections 设计）

Payload 的「集合（Collection）」即数据表 + 后台表单 + 校验三合一。以下为 TypeScript 配置示意（`src/collections/*`）。

```ts
// Products —— 商品（含标签 Blocks、客服关系、动态状态字段）
import type { CollectionConfig } from 'payload'

export const Products: CollectionConfig = {
  slug: 'products',
  admin: { useAsTitle: 'title', defaultColumns: ['title', 'published', 'availableToday', 'sortOrder'] },
  access: {
    read: () => true,                 // 前台公开读
    create: isAdmin, update: isAdmin, delete: isAdmin,
  },
  hooks: { afterChange: [revalidateProduct], afterDelete: [revalidateProduct] },
  fields: [
    { name: 'title', type: 'text', required: true },                 // 商品标题
    { name: 'description', type: 'richText' },                       // Lexical 富文本描述
    { name: 'coverImage', type: 'upload', relationTo: 'media', required: true },
    { name: 'images', type: 'array', fields: [                       // 详情多图
      { name: 'image', type: 'upload', relationTo: 'media' },
    ]},
    { name: 'published', type: 'checkbox', defaultValue: false, index: true },      // 上架开关
    { name: 'availableToday', type: 'checkbox', defaultValue: false },             // 今日可接单（开关）
    { name: 'availableTodayText', type: 'text', defaultValue: '今日可接单' },        // 可接单状态文案（文字自定义）
    { name: 'statusText', type: 'text' },                            // 其他自定义状态文字，如"已约满"
    { name: 'category', type: 'relationship', relationTo: 'categories' },           // 商品类型（可复用分类，前台可筛选）
    { name: 'sortOrder', type: 'number', defaultValue: 0, index: true },           // 排序权重
    { name: 'tags', type: 'blocks', blocks: [TextTagBlock, IconTagBlock, ColorBlockTagBlock, LinkTagBlock] },
    { name: 'contacts', type: 'relationship', relationTo: 'contacts', hasMany: true },
  ],
}
```

```ts
// Categories —— 商品类型 / 分类（可复用，驱动前台 /c/[slug] 分类页与导航）
export const Categories: CollectionConfig = {
  slug: 'categories',
  admin: { useAsTitle: 'name' },
  access: { read: () => true, create: isAdmin, update: isAdmin, delete: isAdmin },
  hooks: { afterChange: [revalidateList], afterDelete: [revalidateList] },  // 改名/删除 → 前台同步
  fields: [
    { name: 'name', type: 'text', required: true },                  // 类型名（文字自定义）
    { name: 'slug', type: 'text', index: true },                     // 可选，URL 筛选用
    { name: 'sortOrder', type: 'number', defaultValue: 0, index: true },
  ],
}

// Customers —— 前台客户账号（第二个 auth 集合，与管理员 users 完全隔离）
export const Customers: CollectionConfig = {
  slug: 'customers',
  auth: { loginWithUsername: { allowEmailLogin: false, requireEmail: false } }, // 用户名登录，不强制邮箱
  admin: { useAsTitle: 'username' },
  access: {
    admin:  () => false,                                  // 客户不得进入 /admin
    create: () => true,                                   // 公开注册
    read:   ({ req }) => isAdmin({ req }) || { id: { equals: req.user?.id } }, // 仅本人/管理员可读
    update: ({ req }) => ({ id: { equals: req.user?.id } }),                   // 仅改自己
    delete: ({ req }) => isSuperAdmin({ req }) || { id: { equals: req.user?.id } },
  },
  fields: [
    { name: 'nickname', type: 'text' },                   // 展示名（username 由 auth 自动提供）
  ],
}

// Favorites —— 收藏（customer ↔ product 多对多，仅增删，不可改）
export const Favorites: CollectionConfig = {
  slug: 'favorites',
  access: {
    admin:  isAdmin,                                                  // 后台可只读查看
    create: ({ req }) => req.user?.collection === 'customers',        // 仅登录客户
    read:   ({ req }) => isAdmin({ req }) || { customer: { equals: req.user?.id } },
    delete: ({ req }) => ({ customer: { equals: req.user?.id } }),    // 取消收藏（仅本人）
    update: () => false,
  },
  hooks: {
    // 归属强制为当前登录客户，杜绝替他人收藏
    beforeValidate: [({ req, data }) => ({ ...data, customer: req.user?.id })],
  },
  fields: [
    { name: 'customer', type: 'relationship', relationTo: 'customers', required: true, index: true },
    { name: 'product',  type: 'relationship', relationTo: 'products',  required: true, index: true },
  ],
  indexes: [{ fields: ['customer', 'product'], unique: true }],       // 防重复收藏
}
```

### 4.1 设计要点

- **标签用 `blocks`**：每种标签类型是一个 Block（`text` / `icon` / `colorBlock` / `link`）。新增类型 = 加一个 Block 定义 + 前台一个渲染分支，不改表结构、不改主流程。
- **`Contact.type` + 条件字段**：`value` 与 `qrImage` 按 `type` 条件显示，后台表单自适应。客服与商品**多对多**（`relationship hasMany`），一个客服可绑多商品。
- **`statusText` 独立于 `published` / `availableToday`**：开关控制可见/可制作，文字由管理员自由输入，互不耦合。
- **索引**：`published`、`sortOrder` 加 `index`，前台"已上架按序排列"查询走索引，性能稳定。
- **类型自动生成**：`payload generate:types` 产出 `payload-types.ts`，前后台共用，schema 改了编译期即报错，避免"后台改字段 → 前台白屏"。
- **商品类型用 `categories` 关系（v1.2）**：「商品类型」是**可复用分类**而非每商品自填文本——多商品共用同一类型、文案统一，前台据此生成 `/c/[slug]` **分类页与导航**（§7.2 / 决策 #5）。与自由排列的 `tags` Blocks 互补：`tags` 偏装饰/灵活，`category` 是结构化归类。
- **`availableToday` + `availableTodayText`（v1.2）**：布尔开关控制「今日可接单」，显示文案由 `availableTodayText` 自定义（默认「今日可接单」），在卡片上作为状态标签呈现（设计规范 §6.2）；与 `statusText`（如「已约满」）解耦。
- **两套 auth 集合隔离（v1.2）**：管理员 `users` 与客户 `customers` 各自 `auth`，会话/cookie/权限互不相通。凡涉及写入归属或读取约束（尤其 `favorites`），access 函数必须按 `req.user.collection`（`'users'` / `'customers'`）区分来源，避免越权。
- **`favorites` 仅增删 + 归属强制 + 唯一索引（v1.2）**：`update:false` 让收藏只增删；`beforeValidate` 把 `customer` 强制写为当前登录客户，杜绝替他人收藏；`(customer, product)` 唯一索引防重复收藏。

## 5. 配置驱动渲染引擎（项目灵魂）

标签"后台高度自定义"决定了不能写死，必须做成**配置驱动的分发渲染系统**，联系方式同理。**数据源头是 Payload 字段定义**（Blocks / select + 条件字段），前台只负责按 type 分发渲染。

### 5.1 前后台共用契约 = 自动生成类型

Payload 字段定义即 schema，`payload generate:types` 自动产出 TypeScript 类型（含标签的 Block 联合、`Contact.type` 联合），前台直接引用，无需手写 Zod。如需对前台输入做额外校验，可在边界处补 Zod，但**契约源头是 Payload**。

### 5.2 分发组件

```tsx
// <TagRenderer> —— 按 blockType 分发，新增类型只加一个 case
function TagRenderer({ tag }: { tag: Product['tags'][number] }) {
  switch (tag.blockType) {
    case 'text':       return <TextTag {...tag} />;
    case 'icon':       return <IconTag {...tag} />;
    case 'colorBlock': return <ColorBlockTag {...tag} />;
    case 'link':       return <LinkTag {...tag} />;
    default:           return null; // 未知类型安全降级
  }
}

// <ContactRenderer> —— 按 contact.type 分发
function ContactRenderer({ contact }: { contact: Contact }) {
  switch (contact.type) {
    case 'wechat':   return <WeChatId {...contact} />;
    case 'wechatQr': return <WeChatQr {...contact} />;
    case 'qq':       return <QQId {...contact} />;
    case 'qqQr':     return <QQQr {...contact} />;
    default:         return null;
  }
}
```

### 5.3 联系方式交互设计（引流核心）

| 形态 | 交互 |
|---|---|
| **微信号** (`wechat`) | 一键复制（`navigator.clipboard`）+ "已复制，去微信添加"提示 |
| **微信二维码** (`wechatQr`) | 点击放大；移动端长按保存到相册；PC 端悬停放大 |
| **QQ 号** (`qq`) | 一键复制；深链 `mqqwpa://im/chat?chat_type=wpa&uin={qq}` 直接唤起加好友 |
| **QQ 二维码** (`qqQr`) | 点击放大查看 |

呈现用精致弹层（Modal / Drawer）+ Motion 动效。每个商品可在后台绑定不同客服与展示哪些联系方式。

> 备注：微信/QQ 主要面向华人客户。未来若有海外非华人客户，这套配置化联系模块可无缝扩展 WhatsApp / Telegram，无需改架构。

## 7. 前台（访客，浏览免登录）

> **视觉设计**：本节定义结构与行为；具体配色 / 字体 / 间距 / 组件视觉 / 动效参数见独立文档 [前端设计规范.md](前端设计规范.md)（杂志编辑风 · 双主题 · 玫瑰粉强调色）。

### 7.1 画廊列表页 `/`

- 响应式网格：PC 多列、平板 2–3 列、移动 1–2 列（Tailwind 断点）。
- 用 **Payload Local API** 在 Server Component 读取 `published = true` 的商品，按 `sortOrder` 排序。
- 封面图懒加载 + 骨架屏 + 模糊占位（避免 CLS 布局抖动）。
- 顶部 **分类导航条**（见 §7.2）：全部 / 今日可接单 / 各商品类型，作为分类页面的入口。
- 卡片展示：封面、标题、商品类型（分类）、状态文字徽标、"今日可接单"高亮标识。
- Motion 网格入场动效。

### 7.2 分类导航与分类页面（需求①核心）

「今日可接单」「商品类型」是**可浏览的分类（各有独立页面）**，而非仅卡片标签——访客点导航进入对应分类页，只看该类商品。

- **分类导航条**（组件 `components/category/`）：横向列出 `全部` / `今日可接单` / 各**商品类型**（来自 `categories` 集合，按 `sortOrder`）。放列表页顶部 / 页眉，编辑风眉标样式（[前端设计规范.md](前端设计规范.md)）。
- **商品类型分类页** `/c/[slug]`：Local API 按 `category` 关系过滤 `published` 商品，复用画廊网格与卡片；页标题取分类名；ISR 静态缓存。
- **今日可接单分类页**：保留 slug `today`（`/c/today`），按 `availableToday = true` 过滤；页标题取 `availableTodayText`（默认「今日可接单」）。它是**虚拟分类**（由每日变动的布尔驱动，非 `categories` 条目）；`categories` 集合须避免使用 `today` 作为 slug。
- **即时生效**：商品 `category` / `availableToday` 改动、分类改名 / 删除，都要 revalidate 受影响的分类页（`/`、`/c/[slug]`、`/c/today`）——见 §8.4 / §8.5。
- 卡片上的「商品类型」「今日可接单」小标仅作辅助展示，可点跳到对应分类页。

### 7.3 商品详情页 `/p/[id]`

- 大图轮播 / 多图展示（`next/image` 优化）。
- 标签区：`<TagRenderer>` 渲染该商品全部标签 Block。
- 富文本区：渲染 Lexical 描述。
- 状态区：显示「今日可接单」状态（开关 `availableToday`，文案取 `availableTodayText`）与 `statusText`；商品类型可点跳对应分类页。
- 联系方式区：`<ContactRenderer>` + 精致弹层，扫码 / 复制 / 唤起 QQ。
- View Transitions 实现列表↔详情丝滑过渡。

### 7.4 渲染与缓存

- ISR：`export const revalidate = 3600`（或更长），配合 Payload 保存时 on-demand `revalidatePath`。列表页、详情页、**分类页**均走 ISR。
- 全球 CDN 边缘缓存，几万访客零服务端压力。
- **会员功能不破坏静态缓存**：画廊 / 详情 / 分类页**内容**仍走 ISR；「当前用户是否已收藏」为每用户状态，由客户端登录后异步拉取并 hydrate；个人中心 `/me` 为每用户**动态页**（`force-dynamic` / 不缓存）。即「公共内容静态、私有状态动态」分离，互不污染。

### 7.5 客户账号（注册 / 登录，可选）

- **注册** `/register`：用户名 + 密码（前台表单 → Payload `POST /api/customers`）。最小可用，无邮箱验证 / 找回密码（决策 #8）。
- **登录** `/login`：→ `POST /api/customers/login`，Payload 下发 httpOnly cookie（与管理员会话隔离）。
- **登录态读取**：Server Component 用 `payload.auth({ headers })` 取当前 `customer`（封装于 `lib/customer.ts`）；客户端可用 `GET /api/customers/me`。
- **页眉入口**：未登录显示「登录 / 注册」；已登录显示「个人中心 / 退出」。浏览全程不强制登录。

### 7.6 个人中心 `/me`

- 仅登录可见，未登录重定向 `/login`。
- 展示**当前用户收藏的商品**：Local API 查 `favorites`（`where customer = 当前用户`）并 populate `product`，复用画廊卡片渲染。
- 提供退出登录入口。每用户动态渲染，不走 ISR / CDN 缓存。

### 7.7 收藏交互（仅收藏）

- 商品卡片 / 详情页放**收藏按钮**（client，心形 toggle）。
- 未登录点击 → 引导去 `/login`。
- 已登录：`POST /api/favorites`（收藏）/ `DELETE /api/favorites/:id`（取消）；归属由 `beforeValidate` 强制为当前用户，唯一索引防重复。
- 乐观更新 + 失败回滚；收藏态在页面加载后由客户端按当前用户拉取一次（不写入静态页）。
- **仅收藏**，不下单、不支付（与 §1.1 一致）。

## 8. 后台（Payload Admin，`/admin`，内置鉴权保护）

### 8.1 鉴权

- 后台 `/admin` 由 Payload 内置登录保护（`users` 集合 `auth: true`，密码自动哈希 + 会话管理），**无需 Auth.js，也无需自写 middleware**。
- 前台 `(site)` 完全不受影响，访客免登录。

### 8.1.1 多管理员与超管（决策 #2）

- 支持**多个管理员账号**，每位工作人员各自登录。
- `Users.role` 区分 `admin`（管理商品/标签/客服）与 `superadmin`（额外可增删账号、改角色）。
- 用 Payload **access control** 实现：
  - `Users.create / delete` → 仅 `isSuperAdmin`；
  - `Users.update` → 超管管全部，普通管理员仅能改自己的资料与密码（返回 `{ id: { equals: req.user.id } }` 约束）；
  - `role` 字段 `access.update` → 仅超管可改，防止管理员自升级。
- 账号管理直接复用 Payload 后台里的 `Users` 集合视图（超管可见、可增删）。

### 8.2 商品管理（`products` 集合）

- **列表视图**：Payload 自带搜索、排序、过滤、分页。
  - 字段级编辑：**上架开关**、**今日可接单开关**、**可接单文案**、**状态文字**、**商品类型（分类）**在编辑页直接改。
  - "今日可接单"**一键批量重置**：用列表的**批量编辑（bulk edit）**勾选多行统一置否；或加一个自定义后台按钮调 Local API（`payload.update` with `where`）每天营业前清零。
- **新建 / 编辑**：Payload 自动生成表单 + 字段校验。
  - 字段：标题、Lexical 富文本描述、封面图、多图上传、排序、**商品类型（分类）**、标签 Blocks、绑定客服。

### 8.3 标签管理 / 客服管理

- 标签：在商品编辑页以 **Blocks** 维护（选 Block 类型、填 `label` / 图标 / `style` / `url`）。
- 客服：独立 `contacts` 集合维护微信号 / 微信二维码 / QQ 号 / QQ 二维码，可被多个商品通过关系复用。

### 8.4 写操作与即时生效

- 所有写操作经 Payload 集合（自动字段级校验）。
- 保存/删除后由集合 `afterChange` / `afterDelete` hook 调用 `revalidatePath`：`/`、`/p/${id}`，以及受影响的**分类页**（该商品 `category` 对应的 `/c/[slug]`；`availableToday` 变动时 `/c/today`）。管理员一改，访客秒级看到最新状态。

### 8.5 分类管理（`categories` 集合）

- 管理员维护「商品类型」列表（名称即文案、可排序、含 `slug`），商品在编辑页选择所属分类；分类即前台 `/c/[slug]` 分类页与导航项。
- 分类 `afterChange` / `afterDelete` 触发 `revalidatePath('/')` 与 `/c/[slug]`，改名 / 删除前台（含分类页与导航）同步。

### 8.6 客户账号与收藏（后台视角）

- `customers`（前台客户）与 `favorites`（收藏）在 `/admin` 对管理员**可见可查**，便于支持与排查；**客户账号不得登录 `/admin`**（`customers.access.admin` 返回 `false`）。
- 客户删除等敏感操作受 access 约束（本人或超管）。
- 两套鉴权（管理员 `users` / 客户 `customers`）会话、cookie、权限完全隔离；access 函数需按 `req.user.collection` 区分来源，防越权（§4.1）。

## 9. 图片与部署

### 9.1 图片

- 上传到对象存储：**Supabase Storage**（S3 兼容），通过 `@payloadcms/storage-s3` 适配，Payload `media` 集合负责上传与记录。
- 多尺寸：Payload `imageSizes` 自动生成缩略图（列表）与大图（详情），降低流量。
- 前台用 `next/image`：自动 WebP/AVIF、响应式 srcset、懒加载、模糊占位；`next.config.ts` 配置存储域名 `remotePatterns`。

### 9.2 部署

- 平台：Vercel（全球边缘 CDN，原生支持 ISR / next/image / Payload 3 serverless）。
- 数据库：Supabase 托管 Postgres。
- 存储：Supabase Storage（serverless 文件系统是临时的，**必须**用对象存储 + storage adapter）。
- 迁移：Payload 用 `payload migrate`，构建时执行；首次 `payload migrate:create` 生成迁移。新增集合（`categories` / `customers` / `favorites`）与 `products` 新字段（`category` / `availableTodayText`）需各自 `payload migrate:create` 增量迁移后再部署。
- 环境变量：`DATABASE_URI`、`PAYLOAD_SECRET`、`NEXT_PUBLIC_SERVER_URL`、Supabase Storage 的 S3 端点/密钥（`S3_*`）等（见 `.env.example`）。
- 验证：全球访问速度、管理员改状态 → 前台 revalidate 生效、联系方式交互（复制/扫码/QQ 唤起）、多管理员/超管权限边界。

## 附：术语

- **Payload CMS**：Next.js 原生的开源 headless CMS，集合定义即数据表 + 后台表单 + 校验，自带鉴权与上传。
- **Local API**：Payload 的服务端直连 API（`payload.find` 等），无 HTTP 跳转，适合 Server Components。
- **Lexical**：Payload 默认的富文本编辑器框架。
- **Blocks**：Payload 的区块字段，用于结构化、可自定义的内容（本项目用于标签类型分发）。
- **access control**：Payload 基于函数的权限控制，可返回布尔或查询约束，按 `req.user.role` 控制读写。
- **storage adapter**：把 Payload 上传对接到外部对象存储（如 `@payloadcms/storage-s3` → Supabase Storage / R2）。
- **ISR**：增量静态再生，静态页可按时或按需刷新。
- **on-demand revalidation**：按需触发某页缓存刷新（`revalidatePath`）。
- **CLS**：累积布局偏移，衡量页面加载时元素跳动的指标，越低越好。
- **auth 集合**：Payload 中开启 `auth` 的集合，自带登录 / 密码哈希 / 会话。本项目有两套——管理员 `users` 与前台客户 `customers`，互相隔离（§4.1）。
- **loginWithUsername**：Payload auth 选项，用用户名（而非邮箱）登录，契合「自设用户名+密码」（决策 #7–8）。
- **favorites（收藏关联集合）**：`customer ↔ product` 多对多，仅增删、唯一索引防重复，个人中心据此展示（§4 / §7.6–7.7）。
