## 2. 开发路线图（里程碑）

按依赖顺序排成 5 个里程碑。**M1 是一切前台工作的前置**（没有类型和 Local API，前台无法类型安全地读数据）。

```
M1 数据层打通 ──▶ M2 配置渲染引擎 ──▶ M3 前台页面 ──▶ M4 即时生效 & 图片 ──▶ M5 部署上线
  (后端为主)         (前端为主)          (前端为主)        (全栈)            (后端为主)
```

| 里程碑 | 目标 | 主导端 | 依赖 |
|---|---|---|---|
| **M1** | 数据库迁移 + 类型生成 + Local API 单例 + seed 数据 | 后端 | — |
| **M2** | TagRenderer / ContactRenderer 分发组件 + UI 依赖接入 | 前端 | M1（类型） |
| **M3** | 画廊列表页 `/` + 商品详情页 `/p/[id]` | 前端 | M1、M2 |
| **M4** | revalidate 联调 + 今日可制作批量重置 + 图片优化全链路 | 全栈 | M3 |
| **M5** | Supabase + Vercel 部署、环境变量、构建迁移、验收 | 后端 | M1-M4 |

## 3. 任务分解（按里程碑）

## 4. 前端 / 后端 责任划分总览

### 【后端】（Payload / 数据 / 部署）
- 集合定义与演进（Products / Contacts / Media / Users）— **已完成**
- 鉴权、多管理员 / 超管、字段级 access — **已完成（待运行时验证）**
- 迁移、`generate:types`、seed — **M1 待办**
- revalidate hook 联调、今日可制作批量重置 — **M4 待办**
- Supabase / Vercel / 环境变量 / 构建迁移 / 验收 — **M5 待办**

### 【前端】（`(site)` / 组件 / 交互）
- shadcn/ui + Motion + lucide 接入 — **M2 待办**
- TagRenderer / ContactRenderer 及子组件 + 联系交互 — **M2 待办**
- 画廊列表页 + 商品详情页 + 网格 / 卡片 / 动效 / View Transitions — **M3 待办**
- `next/image` 图片优化全链路 — **M4 待办**

### 【全栈契约】（前后端交界，易出错处）
- `payload-types.ts`：schema 即类型，前后台共用，改字段编译期报错（M1-2）
- Local API 读取链路：Server Component → `getPayload` → `payload.find`（M1-4 / M3-2）
- ISR + on-demand revalidate：静态缓存 ↔ hook 刷新（M3-7 / M4-1）
- 图片 `remotePatterns` + `imageSizes`：上传尺寸 ↔ 前台取用（M4-3/4/5）

## 5. 关键风险与注意事项

1. **Next.js 16 是非常规版本**（见 [AGENTS.md](../AGENTS.md)）：写任何代码前先查 `node_modules/next/dist/docs/` 对应指南，API / 约定可能与训练数据不同，留意废弃提示。
2. **M1 是硬前置**：没有 `payload-types.ts` 与 Local API 单例，M2/M3 无法类型安全推进，不要跳着做。
3. **本地→生产存储切换**：`payload.config.ts` 已用 `hasS3` 条件区分（本地走磁盘、生产走 Supabase Storage）；M4-3 的 `remotePatterns` 两种来源都要覆盖。
4. **Serverless 文件系统是临时的**：生产**必须**用对象存储，本地磁盘上传仅限开发（[开发文档.md §9.2](开发文档.md)）。
5. **权限边界需运行时验证**：access 函数已写，但「管理员不能自升级 role」「普通管理员只能改自己」要在 M1-5 后实际点一遍。
6. **每阶段闸门**：每个里程碑结束跑 `next build` + `tsc` + `payload generate:types`，全绿再进下一步（[开发文档.md §10](开发文档.md) 收尾约定）。
7. **动效无障碍**：所有 Motion / View Transitions 尊重 `prefers-reduced-motion`（[前端设计规范.md](前端设计规范.md)）。

## 6. 建议执行顺序（下一步）

当前应立即开始 **M1**，按 M1-1 → M1-7 顺序推进：先把数据库迁移、类型生成、Local API 单例和 seed 数据打通，让前台具备可读、可类型推导的数据基础，再进入 M2 的渲染引擎。
