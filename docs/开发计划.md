## 5. 关键风险与注意事项

1. **Next.js 16 是非常规版本**（见 [AGENTS.md](../AGENTS.md)）：写任何代码前先查 `node_modules/next/dist/docs/` 对应指南，API / 约定可能与训练数据不同，留意废弃提示。
2. **两套 auth 集合不可混淆**（M7 核心风险）：`favorites` 的 `create/read/delete` access 必须按 `req.user.collection` 判定，否则可能管理员请求与客户请求互相越权；归属用 `beforeValidate` 强制写 `customer`，不能信任前端传入。
3. **收藏不可破坏静态缓存**（M7）：画廊/详情仍 ISR，「是否已收藏」只能客户端按当前用户拉取后 hydrate；个人中心 `/me` 用 `force-dynamic`，切勿被 CDN 缓存到他人收藏。
4. **客户不得进入 `/admin`**（M7）：`customers.access.admin` 必须返回 `false`，并在运行时点验。
5. **重复收藏防护**（M7）：`(customer, product)` 唯一索引 + 前端乐观更新失败回滚，避免脏数据。
6. **分类页/今日可接单页要 revalidate**（M6）：`categories` 改名/删除须 revalidate `/` 与 `/c/[slug]`；商品 `category`/`availableToday` 改动须 revalidate 对应 `/c/[slug]` 与 `/c/today`，否则分类页/导航显示滞后；删除分类时商品 `category` 关系的悬空处理也要验证。`/c/today` 为保留 slug，分类不可占用。
7. **schema 变更须迁移 + 类型生成**（M6/M7）：新增集合/字段后 `payload migrate:create` + `generate:types`，再 `next build`；切勿手改数据库。
8. **本地→生产存储切换**：`payload.config.ts` 已用 `hasS3` 条件区分；线上必须对象存储（[开发文档.md §9.2](开发文档.md)）。
9. **每里程碑闸门**：结束跑 `next build` + `tsc` + `payload generate:types`，全绿再进下一步。
10. **动效无障碍**：所有 Motion / View Transitions（含收藏按钮反馈）尊重 `prefers-reduced-motion`（[前端设计规范.md](前端设计规范.md)）。

## 6. 建议执行顺序（下一步）

M1–M5 已完成。本轮新需求按下列顺序推进：

1. **先做 M6**（体量小、直接补齐需求①分类页缺口）：M6-1 → M6-10，先后端（分类集合 + 商品字段 + 迁移 + seed），再前端（分类导航 + 分类页 `/c/[slug]` + 今日可接单页 `/c/today`），最后扩展 revalidate 覆盖分类页。
2. **再做 M7**（会员体系，体量大）：M7-1 → M7-11，先后端（customers / favorites 集合 + 迁移 + `lib/customer.ts`），再前端（登录注册 + 收藏按钮 + 个人中心），最后运行时权限边界验证。
3. M6 / M7 各自结束跑闸门（`next build` + `tsc` + `generate:types`），并补充[前端设计规范.md](前端设计规范.md) 中新增组件（分类导航 / 分类页、收藏按钮、登录/注册、个人中心）的视觉细则。

> M6 与 M7 相互独立，如需并行，注意两者都改 `payload.config.ts` 与会触发 `generate:types`，合并迁移时按时间戳顺序执行。
